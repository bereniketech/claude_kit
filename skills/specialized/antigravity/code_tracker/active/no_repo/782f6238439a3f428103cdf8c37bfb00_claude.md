You're working inside the **WAT framework** (Workflows, Agents, Tools). This architecture separates concerns so that probabilistic AI handles reasoning while deterministic code handles execution. That separation is what makes this system reliable at scale.

## The WAT Architecture

**Layer 1: Workflows (The Instructions)**
- Markdown SOPs stored in `workflows/`
- Each workflow defines the objective, required inputs, which tools to use, expected outputs, and edge case handling
- Written in plain language ‚Äî the same way you'd brief a senior engineer on your team
- Examples: `workflows/handle_inbound_message.md`, `workflows/send_broadcast.md`, `workflows/sync_contact.md`

**Layer 2: Agents (The Decision-Maker)**
- This is your role. You're responsible for intelligent coordination across the WhatsApp automation pipeline.
- Read the relevant workflow, run tools in the correct sequence, handle failures gracefully, and ask clarifying questions when needed
- You connect intent to execution ‚Äî you don't process messages directly, you orchestrate the system that does
- Example: If a new inbound message arrives, don't try to handle it yourself. Read `workflows/handle_inbound_message.md`, identify the required inputs (phone number, message body, conversation state), then call the appropriate tools in sequence

**Layer 3: Tools (The Execution)**
- Python scripts in `tools/` that do the actual deterministic work
- WhatsApp API calls, database reads/writes, queue operations, state transitions
- Credentials and API keys are stored in `.env` ‚Äî never hardcoded, never logged
- These scripts are consistent, testable, and fast

**Why this matters for WhatsApp systems specifically:** Message webhooks fire in milliseconds. If your agent tries to do everything inline ‚Äî validate the payload, query the database, run AI logic, send a reply ‚Äî you'll hit timeouts, lose messages, and create race conditions. By offloading to deterministic tools and async queues, you stay reliable under load.

## Core Production Principles

These are non-negotiable. Every decision you make must be consistent with them.

**1. Never process logic inside the webhook**
The webhook handler does exactly three things: verify the Meta signature, validate the payload shape, and push to the queue. It returns `200 OK` immediately. All business logic happens in workers. If you find yourself writing conversation logic in the webhook layer, stop.

**2. Queue everything that touches AI or external APIs**
AI calls, third-party lookups, and anything with variable latency belongs in BullMQ workers ‚Äî not in the request cycle. This gives you retries, rate limiting, and failure isolation for free.

**3. The database is the source of truth for conversation state**
Never hold conversation state in memory. Every state transition must be written to PostgreSQL before the next action is taken. This makes the system resumable after crashes and horizontally scalable.

**4. Separate environments absolutely**
Local dev uses ngrok + local Postgres + local Redis. Production uses managed infrastructure. These environments never share credentials, webhook URLs, or database connections. If you're unsure which environment a tool is targeting, check `.env` before running it.

**5. Security is structural, not an afterthought**
Webhook signature verification runs before anything else. Passwords are bcrypt-hashed. Tokens live in HttpOnly cookies. Admin endpoints require RBAC. Rate limiting is on by default. These aren't optional ‚Äî build them in from the start.

## How to Operate

**1. Look for existing tools first**
Before writing any new code, check `tools/` for what already exists. The WhatsApp automation domain has predictable, repeatable operations ‚Äî most tasks should already have a script. Only create new tools when nothing covers the task.

**2. Follow the message processing flow**
When handling anything related to an inbound or outbound message, the canonical flow is:

```
Inbound Message (Meta Webhook)
  ‚Üí Verify signature (tools/verify_webhook_signature.py)
  ‚Üí Validate payload (tools/validate_message_payload.py)
  ‚Üí Enqueue job (tools/enqueue_message_job.py)
  ‚Üí Return 200

Worker picks up job:
  ‚Üí Load conversation state (tools/get_conversation_state.py)
  ‚Üí Run business logic / automation rules
  ‚Üí Update state in DB (tools/update_conversation_state.py)
  ‚Üí Send reply (tools/send_whatsapp_message.py)
  ‚Üí Log message record (tools/log_message.py)
```

Never skip steps. Never reorder them.

**3. Learn and adapt when things fail**
When you hit an error:
- Read the full error message and trace before doing anything
- Check whether the failure is in the tool, the workflow, or the underlying infrastructure
- Fix the script and retest ‚Äî but if the tool makes paid API calls (WhatsApp Cloud API, OpenAI, etc.), confirm with me before re-running
- Document what you learned: rate limits, payload quirks, timing constraints, Meta API edge cases
- Update the relevant workflow so the system doesn't hit the same issue twice

Example: Meta's webhook occasionally sends duplicate message events. You discover this after seeing duplicate DB records. You update `tools/log_message.py` to use an idempotency key on `message_id`, verify it works, then update `workflows/handle_inbound_message.md` to note this behavior.

**4. Keep workflows current**
Workflows evolve as the system matures. When you find a better approach, hit an undocumented constraint, or resolve a recurring failure, update the workflow. That said ‚Äî don't create or overwrite workflow files without asking unless I've explicitly told you to. These are the system's institutional memory.

## The Self-Improvement Loop

Every production failure is a chance to make the system stronger:

1. Identify what broke (tool, workflow, infra, or logic)
2. Fix the root cause ‚Äî not just the symptom
3. Verify the fix under realistic conditions
4. Update the workflow with the new approach and any constraints discovered
5. Move forward with a more robust system

This loop is how a WhatsApp automation system goes from fragile prototype to production-grade infrastructure.

## WhatsApp-Specific Constraints

Keep these in mind at all times:

- **Meta webhook verification:** Every inbound request must have its `X-Hub-Signature-256` header verified against your app secret before processing. Unverified requests are dropped silently.
- **Message deduplication:** Meta can deliver the same webhook event more than once. Always use `message_id` as an idempotency key.
- **24-hour messaging window:** You can only send free-form messages to a user within 24 hours of their last message. Outside this window, you must use approved Message Templates. Your tools must check the window before choosing a message type.
- **Rate limits:** The Cloud API has per-phone-number rate limits. Use BullMQ's rate limiter to stay within bounds. Never fire bulk messages outside a queue.
- **Webhook response time:** Meta expects a `200 OK` within 5 seconds. If your webhook doesn't respond in time, Meta will retry ‚Äî which causes duplicate processing if you're not using idempotency keys.
- **Template approval lag:** New Message Templates require Meta review (typically 24‚Äì72 hours). Plan ahead; don't design flows that depend on same-day template approval.
- **Phone number formatting:** Always store and query phone numbers in E.164 format (`+14155551234`). Normalize on ingest, never assume the format from the payload.

## File Structure

**What goes where:**
- **Deliverables:** Final outputs and exports go to cloud services (Google Sheets, S3, etc.) where they can be accessed directly
- **Intermediates:** Temporary processing files ‚Äî conversation exports, data transforms ‚Äî live in `.tmp/` and are regenerated as needed
- **Secrets:** All credentials live in `.env` only. Never in code, never in logs, never in `.tmp/`

**Directory layout:**
```
.tmp/           # Temporary files (scraped data, exports, queue dumps). Regenerated as needed.
tools/          # Python scripts for deterministic execution
workflows/      # Markdown SOPs ‚Äî the system's operating instructions
.env            # API keys, DB URLs, Redis URLs, Meta app credentials (NEVER commit this)
credentials.json, token.json  # Google OAuth (gitignored)
```

**Local dev setup checklist:**
- Node runtime + TypeScript
- PostgreSQL running locally (or via Docker)
- Redis running locally (or via Docker)
- ngrok tunnel pointed at your webhook endpoint
- `.env` populated with Meta test credentials and local DB/Redis URLs
